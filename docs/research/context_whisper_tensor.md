# Whisper-Tensor: Context for Whisper-Agent Integration

> **Status**: Whisper-tensor is pre-alpha and will be months before it can back whisper-agent. This doc grounds future discussions by summarizing what whisper-tensor *is* and what a client (whisper-agent) would actually call.

## What It Is

Whisper-Tensor is a **correctness-first machine-learning inference runtime** written in Rust (Edition 2024). It converts trained models (ONNX, HuggingFace Transformers, GGUF, etc.) into executable computation graphs, lowers them through multiple IR layers (Symbolic → MilliOp), and dispatches execution to pluggable backends (NDArray/CPU, Vulkan/GPU). The goal is transparent inference with precise dtype and shape semantics, full introspection during runs, and no black boxes. **It is an inference engine, not an agent runtime**—it computes forward passes; orchestration is the client's job.

## Current State

**What works today:**
- **Backends**: NDArray (CPU, with optional BLAS) and Vulkan (GPU via SPIR-V, per-op fallback to NDArray). Both pass 2000+ official ONNX conformance tests.
- **Model coverage**: 142 ONNX ops supported. Verified correct (bit-perfect on reference): GPT-2, RWKV-7. Plausible but unverified: Llama 3, Gemma, Qwen, Phi, DeepSeek V2 (LLMs); Stable Diffusion, Flux (image); CogVideoX, HunyuanVideo (video); Whisper, Kokoro, Piper, F5-TTS (speech).
- **WebSocket server** (Axum + Tokio): Accepts SuperGraph execution requests as CBOR, streams tensor introspection reports back. No HTTP endpoint yet.
- **Graph Explorer UI** (WASM/egui): Native and browser frontends for navigating nested computation graphs and inspecting intermediate tensors.

**Honest limitation**: Model loaders were generated by Claude with limited manual review. Shape inference passes; numerically correctness unconfirmed except where noted.

## SuperGraph System

SuperGraph is a **job-level composition layer** above MilliOp graphs. Instead of shipping individual model computations, a client describes a *model ensemble* (LLM + tokenizer load/encode + RNN cache + decoder loop + logit sampling) as a single serializable graph. The server compiles and executes it as one unit.

**Six job interface types:**
- `TextInference` — token sequence in, logits out (LLMs)
- `MultimodalLanguage` — text + vision/audio in, logits out
- `ImageGeneration` — prompts + noise in, image out (diffusion)
- `VideoGeneration` — prompts + noise in, video clip out
- `TextToSpeech` — text in, audio out
- `SpeechToText` — audio in, tokens out

**Graph structure**: Heterogeneous nodes (model execution, tokenization, caching, loops) connected by polymorphic links (tensors, strings, tokenizers, cached state, hashes). Data-driven greedy scheduling executes nodes as inputs become ready. Scan nodes nest SuperGraphs for autoregressive loops with state carried across iterations.

**Why this matters for whisper-agent:** Instead of calling "run this model, then tokenize, then sample," a client commissions a SuperGraph once—a serializable blueprint—and then feeds inputs across WebSocket. The server does the orchestration internally, yielding better performance (fused operations, intelligent batching) and consistent error handling.

See `docs/supergraph.md` in whisper-tensor for full architecture.

## Planned OpenAI-Compatible Completion Endpoint

The user stated that whisper-tensor servers will expose an **OpenAI-compatible HTTP endpoint** (e.g., `/v1/completions`) alongside the richer SuperGraph WebSocket API. **This is planned, not yet implemented** (grep confirms no completion/openai code in `crates/whisper-tensor-server/src/`). A future version would let whisper-agent treat a whisper-tensor server as a drop-in replacement for OpenAI or Ollama clients.

## Interfaces Whisper-Agent Would Actually Call

**Today:**

1. **In-process Rust crate** (direct dependency on `whisper-tensor`):
   - Import a model → `Model::new_from_onnx()`
   - Build a SuperGraph from the model
   - Call `super_graph.eval(&mut data, &mut context)` with prepared inputs
   - Extract outputs from `data`

2. **WebSocket over TCP** (to remote server):
   - Connect to `ws://localhost:PORT/ws`
   - Serialize a `SuperGraphRequest` (SuperGraph, model IDs, tensor/string/hash inputs)
   - Send as CBOR binary frame
   - Receive `SuperGraphResponse` with result or streaming `SuperGraphExecutionReport` updates
   - Optionally subscribe to intermediate tensor introspection during runs

**Future (planned):**

3. **OpenAI-compatible HTTP** (e.g., `POST /v1/completions`):
   - Standard JSON request/response, minimal friction for existing clients
   - Will wrap SuperGraph internally

**Serialization formats:**
- **SuperGraph request/response**: CBOR (binary, compact, sent over WebSocket)
- **Models**: SafeTensors, GGUF, ONNX, HuggingFace Transformers directory layouts

**Default ports/paths** (from server main.rs):
- WebSocket: `/ws` on configured server address
- Static assets: `/` (WASM UI)

## Conventions to Inherit

When integrating with whisper-tensor, whisper-agent should adopt:

1. **Bare repo + worktree layout** — main branch is bare; development happens in worktrees (see `.claude/worktrees/`)
2. **Rust Edition 2024** — baseline for any Rust crate
3. **Error handling**: `thiserror` for domain errors, `anyhow` for context wrapping
4. **Async runtime**: `tokio` at all I/O boundaries; sync inside pure computation
5. **Docs structure**: Long-form architecture docs in `docs/` (not API reference); model-specific notes in roster
6. **CLAUDE.md principles**:
   - Do a small piece well, not everything with shortcuts
   - General paths first, optimizations layered on top
   - Friction signals architectural issues; surface them instead of papering over with wrappers
   - Unfinished beats faked—report progress honestly
   - After corrections, re-derive from the original goal, not patch onto the previous attempt

## Practical Next Steps for Whisper-Agent

1. **Decide on integration point**: in-process Rust API (tight, fast, no network overhead) vs. remote WebSocket (loose coupling, easier versioning). In-process is lower latency; WebSocket is higher isolation.
2. **Model lifecycle**: Pre-load whisper-tensor models at startup and cache them; commissioning a SuperGraph for each request is expensive. Cache the SuperGraph blueprints too.
3. **Streaming inference**: Use the `SuperGraphObserver` trait (already in place) to stream partial results back to the user while a SuperGraph runs (especially for token-by-token text generation).
4. **Error recovery**: Distinguish between transient (OOM, timeout) and permanent (model missing, invalid graph) failures. SuperGraph execution reports include per-node timing and cancel points.
5. **Wait for HTTP endpoint**: The OpenAI-compatible completion endpoint will be easier to test and integrate with existing tools. Current WebSocket + CBOR requires custom marshalling code.
